# Hermes 启动器 — 测试用例库

> **定位**:这是 PM 反馈驱动的回归 + 核心路径用例库,**不是建议清单**。
> 工程师 agent / QA agent 在每次发版前**必须对照这里的用例逐条声明通过**(通过 / 未通过 / 无法本地验证),并附执行证据。
> PM 不再当 QA;PM 只在最后做 5-10 分钟真机抽查。

---

## 1. 目录组织

```
testcases/
├── README.md                 # 本文件:用例库使用指南
├── regression/               # 回归用例 — 跟 bug 走,每修一条 bug 必须新增至少 1 条
├── core-paths/               # 核心用户路径 — P0,覆盖 10 条常见使用场景
└── future/                   # 待补 — Phase 2 摸底任务用,本期暂为空
```

### 1.1 `regression/`

每个文件对应**一个或一组已修过的 bug**。文件名格式:`<字母编号>-<bug-简述>.md`(例:`A-channel-deps-on-demand.md`)。

规则:

- **每修一条 bug,必须新增至少 1 条对应用例**(没有用例的修复 = 没修)
- 用例必须能精确触发原 bug 场景(不是宽泛的"测一下相关功能")
- 用例文件首行必须写明:**触发的 bug 是什么 / 哪个版本引入 / 哪个版本修复**

### 1.2 `core-paths/`

10 条 P0 用户路径,**每个发版前都要对照声明状态**。当前清单:

## 2. 用例格式约定

每条用例都是一个独立的 `.md` 文件。**严禁用占位符或"建议测试一下"这种含糊话**——用例必须可执行、可证伪。

### 2.1 必备字段

```markdown
# TC-XXX: <用例标题>

**优先级**:P0 / P1 / P2
**关联陷阱**:CLAUDE.md #XX, #XX
**适用版本**:vYYYY.MM.DD.X 及以上

## 前置条件
- 操作系统:<具体到 Win10 / Win11 / 版本号>
- hermes 状态:<已装 / 未装 / 装一半>
- .env 内容:<具体配置项 + 值>
- 网络环境:<国内 / 海外 / 离线>
- 其他:<例如 中文用户名 / 含空格路径 / 防病毒拦截 / 高 DPI>

## 测试步骤
1. <具体动作,带可执行命令或点击位置>
2. ...
3. ...

## 预期结果
- 步骤 1 后:<具体看到/产生什么>
- 步骤 2 后:...
- 最终状态:...

## 执行证据(发版前由 agent / PM 填)
- [ ] 步骤 1 截图:_______________
- [ ] 步骤 2 日志:_______________
- [ ] 通过 / 未通过 / 无法本地验证
- [ ] 备注:_______________

## 失败处理
- 如果未通过,定位到代码哪一行:_______________
- 是否需要新建陷阱条目:是/否
```

### 2.2 字段写作要求

- **前置条件**:必须**具体**到能复现。"已装 hermes" 不够,要写"hermes-agent v0.4.9,InstallDir=`%LOCALAPPDATA%\hermes\hermes-agent`,gateway 未运行"
- **测试步骤**:每步必须有动作动词("双击" / "点击 X 按钮" / "粘贴 Y 到输入框")。**禁止**"启动一下"这种含糊话
- **预期结果**:每步预期都要可观察(看到 X UI / 文件 Y 存在 / 进程 Z 在跑)。**禁止**"应该正常工作"
- **执行证据**:**截图必须有具体路径**(例:`testcases/regression/_evidence/B-001.png`)。日志贴尾部 30-50 行即可
- **关联陷阱**:列出本用例如果失败可能命中的 CLAUDE.md 陷阱编号,反向追溯方便

## 3. 谁来跑用例(角色分工)

### 3.1 工程师 agent

- 自检阶段:跑**所有能在 sandbox 里跑通的用例**(代码逻辑 / 配置文件解析 / SelfTest 类)
- 不能跑的(需要真机 / 中文环境 / 国内网络 / 干净 VM)→ **诚实声明盲区**,在用例的"执行证据"章节写"无法本地验证,因为 X"
- **严禁伪造通过**——这是 PM 在 memory `feedback_ui_selftest.md` / `feedback_testing_before_release.md` 里反复强调的红线

### 3.2 QA agent(质检员)

- 对照工程师产出**审用例**(每条用例的执行证据是否充分、盲区声明是否合理、回归用例是否真能触发原 bug)
- 找漏报:工程师没声明的盲区、用例没覆盖的边界场景
- 评分纳入 90 分门槛

### 3.3 PM 真机验收

- 最终 5-10 分钟在真实 Win 机器上**抽查 P0 用例**(不全跑,P0 里挑 3-5 条 PM 关心的)
- 抽查范围每次发版可不一样,但 `core-paths/TC-002 / TC-005 / TC-009` 是**每次必须跑的最小集**

## 4. 如何写新用例

### 4.1 写 regression 用例(每修一条 bug)

1. 工程师 agent 修完 bug 后,在 `regression/` 下新建文件 `<字母编号>-<简述>.md`
2. 字母编号:按时间顺序往后排(A、B、C、D...)
3. 用例必须能**精确触发**原 bug 的场景。例:Bug A 是"`.env` 改了但 watcher 没触发",用例就要写"修改 .env 添加 TELEGRAM_BOT_TOKEN,等 X 秒,验证 gateway 重启"
4. 至少 3 个执行证据点(截图 / 日志 / 命令输出),不止 1 个

### 4.2 写 core-paths 用例(很少新增)

- 只在 PM 提出新的"用户场景"时新增
- 编号续写(下一条是 TC-011)
- 必须满足:**用户实际会做的操作 + 影响 P0 主流程**

### 4.3 修改已有用例

- 只能在"测试步骤 / 预期结果"上**补充**,不能**删除**已有步骤
- 修改要有 git commit,commit message 写明"为什么改"
- 用例的"执行证据"章节每次发版重新填一遍(不是改一次永远复用)

## 5. 反例:这些不是用例

以下表述**不算用例**,工程师写了 QA 必须打回:

- ❌ "建议测试一下安装流程"
- ❌ "确保 gateway 能正常启动"
- ❌ "在中文 Windows 上跑一下应该没问题"
- ❌ "Bug A 已修复,无需重测"

正确的用例:

- ✅ "在 Win11 26200 中文系统,用户名为'张三',双击 `Start-HermesGuiLauncher.cmd`,选默认安装位置,等待安装完成。预期:终端窗口正常关闭,主面板出现'已就绪',无 GBK 报错"

## 6. 与 CLAUDE.md 陷阱清单的关系

- 陷阱清单 = "已经踩过的坑 + 怎么避免"
- 用例库 = "怎么验证坑没复发 + 怎么验证主流程没断"
- 每条用例的"关联陷阱"字段要列陷阱编号,出问题反向追溯
- 新陷阱沉淀时(CLAUDE.md "经验沉淀强制流程"),如果该陷阱有对应用户场景,**必须同步在 `testcases/regression/` 下新增用例**

---

## 7. 用例库的演进规则

- 不允许删用例(陷阱级历史资产,只能加注"已废弃,因为 X")
- 用例数预期增长曲线:Phase 1 上线时 13 条(10 core-paths + 3 regression),后续每发版 +1~3 条
- 6 个月后预计 30~50 条,届时按"主流程 / 渠道 / 中文 / 网络 / 异常恢复"分子目录
- 工程师 agent 启动每个新任务前**必读** `core-paths/` 全部 + 与本任务相关的 `regression/`
