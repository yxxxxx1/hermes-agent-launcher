## 1. 产品目的(Product Goal)

为智简 AI 启动器装上"反馈通道",让用户遇到问题时:
- 不需要找作者社交账号
- 不需要描述"在哪一步"
- 一键就能反馈,自动带上必要的诊断信息

这是"AI 自我升级系统"的第一步基础设施——
让产品能"听到用户的声音"。

## 2. 运营目的(Business Goal)

短期:
- 收集真实用户使用问题(目前我们只能靠用户主动私信)
- 减少"用户装不上但默默放弃"的损失

长期:
- 反馈数据成为"AI 自我升级"的输入
- 反馈频率/类型可量化,作为产品质量指标
- 高频反馈问题 → 自动生成下一轮迭代任务

## 3. 用户视角(User View)

### 当前痛点
- 用户遇到问题不知道怎么反馈
- 用户私信描述"我装不上"但没有诊断信息,作者无法定位
- 大部分用户遇到问题直接放弃,作者不知道
- 高级用户愿意反馈,但要描述很多上下文,嫌麻烦

### 期望状态
- 启动器主界面有显眼的"💬 反馈"按钮
- 点击后弹出简单表单(类别 + 描述)
- 自动附加:版本号 / Windows 版本 / 当前日志摘要
- 一键提交,作者很快能看到

## 4. 成功标准(Success Criteria)

### 必须达成(阻塞条件)
- [ ] 启动器主界面有"💬 反馈"按钮(显眼但不抢戏)
- [ ] 点击按钮弹出反馈对话框
- [ ] 对话框包含:类别下拉(闪退/功能不可用/UI 问题/建议/其他)+ 描述文本框
- [ ] 自动附加诊断信息(版本号 / Windows 版本 / 最后 50 行日志)
- [ ] 用户提交后,反馈能传到作者(具体方式见决策点)
- [ ] 提交后给用户明确反馈("已收到,谢谢")
- [ ] 用户隐私保护(诊断信息脱敏:不包含 API Key / 路径中的用户名)

### 验证条件(真机测试)
- [ ] 在 Windows 10/11 上点击反馈按钮,对话框正常弹出
- [ ] 填写后提交,作者能在 24 小时内看到反馈
- [ ] 提交失败时(网络问题),给用户明确错误提示
- [ ] 重复点击不会触发多次提交(防抖)

### 加分项(可选)

- [ ] 反馈历史:用户能看到自己提交过的反馈

## 8. 决策点(Decision Points)

### 必须 PM 决策的点

#### 决策点 1:反馈传递方式
工程师 Agent 给方案,PM 选:

**方案 A:邮件 SMTP**
- 用户提交 → 启动器调用 SMTP → 发到作者邮箱
- 优点:简单,无后端
- 缺点:作者邮箱地址暴露;SMTP 配置麻烦

**方案 B:飞书表单 Webhook**
- 用户提交 → 启动器 POST 到飞书 Webhook → 自动入飞书表单
- 优点:作者能用飞书统一管理
- 缺点:需要飞书账号/配置 Webhook

**方案 C:Telegram Bot**
- 用户提交 → POST 到 Telegram Bot API → 消息推送到作者 Telegram
- 优点:实时性最好,作者第一时间看到
- 缺点:需要科学上网

**方案 D:GitHub Issues API**
- 用户提交 → 自动在 GitHub repo 创建 Issue
- 优点:开源生态,public 透明
- 缺点:需要用户接受公开;需要保护作者 token

**方案 E:简单的 Cloudflare Worker**
- 用户提交 → POST 到一个 Cloudflare Worker → Worker 转发到作者
- 优点:可控,免费,没单点故障
- 缺点:需要写一个 Worker(简单)

**PM 倾向**:[明天填,基于工程师方案]

#### 决策点 2:反馈按钮位置
- 选项 A:主窗口顶部右上角(最显眼)
- 选项 B:主窗口底部状态栏(不抢戏)
- 选项 C:"关于"对话框内(隐蔽,但不抢戏)

**默认建议**:选项 A(显眼优先)

#### 决策点 3:诊断信息脱敏粒度
- 选项 A:严格脱敏(API Key、用户名、路径全部 mask)
- 选项 B:基础脱敏(只 mask API Key,保留用户名以便诊断)
- 选项 C:用户选择(让用户在提交前看一眼"我们会发送什么")

**默认建议**:选项 C(透明 + 用户控制)
