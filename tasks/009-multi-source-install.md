**任务编号**: 009
**创建日期**: 2026-04-25
**预计耗时**: 6-8 小时
**优先级**: P1
**用作**: 多 Agent 协作流的第一次真实测试

---

## 1. 产品目的(Product Goal)

解决用户最痛的问题:**国内网络环境下装不上**。

当前情况:
- 启动器调用上游 install.ps1 → 走 GitHub raw / PyPI / npm 默认源
- 国内网络下:超时、连接被重置、下载到一半失败
- 用户重试几次 → 放弃 → 流失

目标:**让国内网络环境下的安装成功率从当前(估计 30-50%)提升到 90%+**。

## 2. 运营目的(Business Goal)

短期:
- 直接提升下载转化为"成功使用"的比例
- 减少"装不上"导致的差评 / 退坑
- 减少作者答疑"装不上怎么办"的负担

长期:
- 安装成功率是商业化漏斗的第一个关键节点(装不上 → 不会成为付费用户)
- 这是"工具免费 + 知识付费"模式中"工具"那一端的核心质量
- 装得稳的启动器 = 持续的口碑传播

## 3. 用户视角(User View)

### 当前痛点

**典型用户 A(普通家庭网络)**:
- 双击启动器,点"开始安装"
- 看到"正在下载 Python..."卡了 5 分钟
- 报错"Connection timed out"
- 不知道是自己网络问题还是软件问题
- 关掉启动器,放弃

**典型用户 B(公司网络/有代理)**:
- 类似上面,但还遇到"GitHub 连不上"
- 即使懂技术,也不愿意为了一个工具配代理

**典型用户 C(在校学生宿舍网)**:
- 部分流量被限制
- 大文件下载经常中断
- 没有耐心反复重试

### 期望状态
- 用户不需要任何配置(没有"高级选项"让他选源)
- 安装过程中,如果默认源失败,**自动切换镜像源,不打扰用户**
- 用户看到的是"正在下载...30%...60%...完成",不是"超时,重试"
- 多次切换源对用户透明(可在日志里看到,但不弹窗)

## 4. 成功标准(Success Criteria)

### 必须达成(阻塞条件)

#### 功能层
- [ ] 启动器在调用上游 install.ps1 之前,**预先检测网络环境**
- [ ] 检测结果决定:
  - 海外网络 → 使用上游默认源(GitHub / PyPI / npm 官方)
  - 国内网络 → 使用国内镜像源
- [ ] 国内镜像源至少包含:
  - GitHub:ghproxy.com / kgithub.com / 其他可用镜像
  - PyPI:阿里源 / 清华源 / 豆瓣源
  - npm:淘宝镜像(npmmirror.com)
- [ ] 主源失败时,**自动 fallback 到下一个源**(不让用户看到"失败了")
- [ ] 每个源最多重试 2 次,3 个源都失败才报错

#### 体验层
- [ ] 用户操作路径不变:点"开始安装"→ 等待 → 完成
- [ ] 安装时长不显著增加(< 10% 增加)
- [ ] 失败时的错误提示明确:"已尝试 3 个镜像源,请检查网络"
- [ ] 不要求用户配置代理 / 选择源 / 修改 hosts

### 验证条件(真机测试)

#### 必须测的场景
- [ ] 中国大陆家庭网络(直连)→ 安装成功
- [ ] 中国大陆 + 已配代理(如能模拟)→ 安装成功
- [ ] 海外网络(VPN / 国外服务器)→ 安装成功(用上游源)

#### 测不了的场景(声明盲区,后续监控)
- [ ] 360 等安全软件拦截
- [ ] 公司内网严格防火墙
- [ ] 教育网特殊环境

#### 性能指标
- [ ] 国内安装总时长 ≤ 海外安装时长 × 1.5
- [ ] 单源失败 → fallback → 下一源的切换时间 ≤ 30 秒

### 加分项(可选)

- [ ] 测速逻辑:并行 ping 多个源,选最快的(而不是写死优先级)
- [ ] 进度条:显示当前用的是哪个源(便于诊断)
- [ ] 日志详细记录:用户反馈时,作者能看到"装的时候用了哪个源,失败几次"

## 8. 决策点(Decision Points)

### 必须 PM 决策的点

#### 决策点 1:网络环境检测策略

工程师 Agent 给方案,PM 选:

**方案 A:简单 IP 地理位置检测**
- 调用 ip-api.com 之类的接口,看返回的国家
- 优点:简单
- 缺点:接口可能不稳;有的接口被墙

**方案 B:连通性测试**
- 同时 ping `github.com` 和 `gitee.com`
- 哪个快用哪个对应的源
- 优点:直接反映用户真实网络
- 缺点:需要等待几秒

**方案 C:综合测速**
- 并行测试 5 个源(GitHub 官方 + 4 个国内镜像)
- 用最快的
- 优点:智能,准确
- 缺点:启动慢,逻辑复杂

**方案 D:粗粒度检测**
- 测一次 `raw.githubusercontent.com` 是否可达,5 秒超时
- 可达 → 海外路径;不可达 → 国内路径
- 优点:快、简单
- 缺点:不够精细

**默认推荐**:方案 D(简单可靠)+ 镜像源失败时再 fallback

**PM 倾向**:[等工程师方案后填]

#### 决策点 2:镜像源优先级策略

**方案 A:写死优先级**
- 国内:阿里 > 清华 > 豆瓣
- 改动小,可控

**方案 B:测速选源**
- 每次安装前并行 ping 几个,选最快的
- 用户体验好,但启动慢 5-10 秒

**方案 C:配置文件 + 写死优先级**
- 优先级在 `mirrors-config.json` 里
- 我可以远程更新这个 json,不用发新版

**默认推荐**:方案 A(MVP),v2 升级到方案 C

**PM 倾向**:[等工程师方案后填]

#### 决策点 3:用户透明度

**方案 A:完全透明**
- 进度条只显示百分比
- 用户不知道用了哪个源
- 失败时只说"网络问题"

**方案 B:完全可见**
- 进度条显示"正在通过 [清华源] 下载..."
- 切换源时显示"已切换到 [阿里源]"
- 用户能看到所有过程

**方案 C:分级**
- 默认只显示进度
- 失败/切换时简短提示:"网络较慢,已切换到备用源"
- 不打扰但不隐瞒

**默认推荐**:方案 C(平衡)

**PM 倾向**:[等工程师方案后填]
