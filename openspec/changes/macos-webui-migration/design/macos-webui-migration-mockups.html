macOS WebUI Migration — UI 设计稿

对应 OpenSpec 提案：openspec/changes/macos-webui-migration/proposal.md · 设计稿沿用 LauncherRootView.swift 现有 LauncherPalette token 与圆角/阴影规范，不引入新视觉系统。

本次 UI 改动的范围与原则

范围：阶段卡（3→2）、新增启动进度卡、管理首页简化、错误结果卡、About 加开源 banner、网络阻塞 hero、China network 内联提示、删除"模型/渠道"入口。 不动：色板 / 字体 / 圆角等级 / 按钮规范 / TopBanner / BusyOverlay / 全局背景 glow / DisclosureGroup 折叠样式。

  1. 启动器只做"装 + 启 + 给浏览器入口"，模型/渠道 / 网关配置全部交给 WebUI 承载。
  2. 视觉沿用现有 macOS Big Sur+ 暖米色风格，不做风格大改。
  3. 错误态都给"可操作"的下一步（重试 / 查看日志），不只是显示错误。
  4. 文案简洁、动词开头（沿用现有"开始和 Hermes 对话""检查现在能不能正常用"风格）。
  5. 不引入新颜色 token、不引入新组件库，所有元素都能从现有 SwiftUI primitives 推导。
§1 阶段卡：3 张 → 2 张 §2 启动进度卡（NEW） §3 管理首页 / Dashboard 简化 §4 错误结果卡（4 种代表） §5 About 对话框 §6 网络阻塞 Hero §7 China network 内联提示 §8 删除模型/渠道入口

§2 启动进度卡（NEW LaunchProgressCard）

新增组件 对应 STAGE: 事件协议

改了什么：用户点"开始使用"后，替换当前 hero 区域为竖向 7 步 checklist。每一步四态： waiting 灰 running 旋转 ok 绿勾 failed 红 X。
为什么：从 Bash 子进程读 STAGE:<phase> STATUS=<s> 事件流推动 UI；用户能清楚看到现在卡在哪一步、要等多久；失败时直接给"查看日志"入口。
对应字段：snapshot.launchProgress: LaunchProgress?（idle 时 nil）；LaunchProgress.Phase 8 个 case；LaunchProgress.Status = .running | .ok | .failed。
走完后：设计建议整张卡片折叠为单行成功条（"已启动，浏览器已打开 → http://localhost:8648"），而不是消失，便于用户复查。

7 个阶段顺序： check_node → download_node（仅当系统 Node 缺失或 < 23）→ extract_node → install_webui → start_gateway → wait_gateway_healthy → start_webui → wait_healthy。 系统已有 Node ≥ 23 时跳过 download/extract 两步，直接显示 5 步可见。

§3 管理首页 / Dashboard 简化

LauncherRootView.swift L617–650 / L652–747

改了什么：删除 Hero 区下方的"当前模型/渠道/网关"pill 与"换一个 AI 模型 / 接收消息通知"两个 SimpleActionRow；只保留"打开浏览器对话"主入口。维护项（模型/网关/工具/卸载）统一收到下方 DisclosureGroup。

§4 错误结果卡（4 种代表）

LauncherRootView.swift L196–294 (ResultOverlay)

改了什么：沿用现有 ResultOverlay 视觉（44px 图标方块 + tone 色 + 主/次 / 知道了三按钮），按 proposal §4.3 错误映射给 4 种代表性 REASON 一种视觉。
为什么：每个错误必须给"可操作的下一步"，不能只显示文字。不再 fall back 到 CLI / 终端对话（按 proposal §7.3 决策），所有失败都给"重试 + 查看日志"两条路。
对应字段：LauncherResultCard { tone, title, message, primaryActionTitle/ID, secondaryActionTitle/ID }。

共同布局：宽 460px / 圆角 28 / 浅背景蒙板（black 12% opacity）/ 顶部 row 含 44px 图标方块 + 标题(22pt bold rounded) + 正文(14pt 1.7 line-height)。底部按钮区主+次+知道了。

A. node_download_failed · 网络问题

!

Node.js 下载失败

网络看起来不太通畅，试了 3 次都没下载成功。可以重试，或者暂时改用本机已经装好的 Node（≥ v23）。

tone: .warning ｜ primaryActionID: "retry-launch" ｜ secondaryActionID: "open-network-docs"

B. npm_install_failed_* · 安装日志入口

!

WebUI 安装失败

npm 退出码 1，可能是镜像源短暂不通。我们已经把详细记录写到日志里，可以查看排查；或直接重试。

tone: .warning ｜ primaryActionID: "retry-launch" ｜ secondaryActionID: "open-install-log" → ~/.hermes/launcher-runtime/install.log

C. webui_failed · 不再 fall back CLI

!

浏览器对话界面没启动起来

WebUI 进程退出了。可以重试，或者打开服务日志看看具体出错的位置。

tone: .warning ｜ 不再有"改用终端对话"按钮（proposal §7.3 决策）｜ secondaryActionID: "open-webui-log" → ~/.hermes-web-ui/server.log

D. health_timeout · 进程在但 /health 不通

!

WebUI 已启动，但响应较慢

进程正在跑，但 30 秒内 /health 还没通。可以再等等，或者停止后重试。

tone: .warning ｜ primaryActionID: "stop-and-retry" ｜ secondary: "open-webui-log" ｜ tertiary: dismiss(再等 30 秒)

§6 网络阻塞 Hero state

平移自 Windows Bug N · 1fbb26a

改了什么：preflight 探测 GitHub / PyPI / npm 三个源都连不上时，替换整个 SetupHero 区域为 hero 错误态。
为什么：让用户立刻知道是网络环境问题、不是程序坏了；给可操作的"重试 / 查看排障文档"路径，避免反复点装。
对应字段：bash STAGE:preflight STATUS=failed REASON=network_blocked DETAIL=github,pypi,npm，Swift 端解析 DETAIL 显示三项探测各自的连通状态 chip。

§8 删除模型/渠道相关入口的影响

LauncherRootView.swift L1031–1051 (managementUsageActions / managementMaintenanceActions)

改了什么：删掉所有"配置模型 / 换 AI 模型 / 接收消息通知 / 配置网关 / 启动网关"等启动器内入口；这些能力下沉到 WebUI 内承载。统一以"在浏览器中打开 WebUI"作为唯一入口。
为什么：启动器与 WebUI 并存时这些入口会混淆用户（究竟改启动器的还是 WebUI 里的？）；现在启动器只暴露"装+启"，避免分裂。
对应字段：managementUsageActions 数组从 5 项减为 3 项；managementMaintenanceActions 数组移除 chat_terminal（不再 fallback CLI），新增 webui-restart / webui-stop / webui-log。
