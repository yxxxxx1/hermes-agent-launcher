Hermes macOS Launcher — 设计稿 v2

基于"启动器只做一件事"原则的彻底简化版。v1 保留作为对照，本版覆盖最终落地形态。

设计原则

启动器的唯一职责是把用户送进浏览器里的 Hermes WebUI。模型、渠道、网关、消息平台、聊天界面 —— 全部在 WebUI 内承载。启动器不参与、不入口、不分流。

核心结构：1 个主视图 × 5 种 Hero 状态。窗口外壳（标题栏 + 主区 + 底栏）始终一致；切换状态只换 Hero 内容。用户心智上是"同一个窗口在变状态"，不是"不同的页面"。

主视图：720 × 560 单一尺寸。不再区分 setup / management 模式。Hero 区使用居中五段式（视觉 / 标题 / 副文 / 主 CTA / 辅助链接）。底栏显示运行时栈与 ⋯ 维护菜单入口。

5 种 Hero 状态：① 未安装 · ② 安装/启动进行中（含 7 子状态）· ③ 运行中 · ④ 出错（含 4 错误变体，无 CLI 兜底）· ⑤ 网络阻塞。状态 1 / 2 在国内网络下 Hero 副文下方追加一条淡黄 inline 提示条，无独立 CTA。

不在首屏出现：维护卡片、模型/渠道入口、"接下来你可能会做的事"列表、设置面板、终端对话兜底。这些要么在 ⋯ 菜单里，要么在 WebUI 内，要么直接不存在。

§2d 状态 2d：简化版 3 阶段进度（面向小白用户，§2c 的人话版）

用户反馈：§2c 的 11 行清单含太多技术名词（Node.js、npm、hermes-web-ui、lark-oapi），普通用户看不懂"装到哪了"。本节简化为 3 大阶段，技术细节默认折叠：
① 准备运行环境（合并 check_node + download_node + extract_node + install hermes-web-ui）
② 配置聊天工具（detail 显示用户配过的平台名：微信、飞书、Telegram、企业微信，不出现 lark-oapi / python-telegram-bot 之类）
③ 启动 Hermes 服务（合并 start_gateway + wait_gateway + start_webui + wait_healthy）
行号显示 第 1 步 / 第 2 步 / 第 3 步，不再用 7/11 这种暗示有 11 步。底部"查看技术日志"小灰字链接默认折叠，本期 mockup 不画展开态（保留 §2c 作为详细技术视图备份，后续可能给 power user）。
